# GHS Provider Branding, Visibility, and Caching Master Policy

This is the unified master policy for GioHomeStudio (GHS).

This document defines how GHS presents intelligence, providers, rendering engines, roles, and caching behavior across the entire system.

This applies to:

- Movie Creator
- Music Creator
- Music Video Creator
- Auto Content Creator
- Image / Video / Music Generation Systems
- Planning Intelligence Systems
- Rendering Engines

This document combines the original branding strategy with the provider visibility architecture and caching safety rules.

The core philosophy must remain intact.

---

# 1. Core Product Rule

GHS is the brand.

Users must experience:

- GHS intelligence
- GHS workflow roles
- GHS quality tiers
- GHS recommendations
- GHS production logic

Not:

- raw vendor identities
- technical provider confusion
- backend naming exposure

Important rule:

**The normal user interface must feel like GHS is doing the work, not external providers.**

---

# 2. Intelligence Layer Visibility Rule (Hidden Layer)

The planning and reasoning system must remain hidden from the user.

This includes:

- story expansion
- missing detail inference
- sound logic
- choreography logic
- scene expansion
- emotional pacing
- atmosphere logic
- technical breakdown
- continuity checks
- production logic

These systems may internally use:

- multiple LLM providers
- multiple reasoning engines
- fallback routing logic
- hybrid orchestration

But users must never be forced to choose these systems directly.

---

# 3. Intelligence Role Presentation

Instead of showing provider names, show functional roles.

Recommended role labels:

- AI Story Director
- AI Technical Director
- AI Quality Reviewer
- AI Music Planner
- AI Visual Director
- AI Continuity Checker
- AI Production Reviewer
- AI Scene Planner
- AI Shot Planner

Important rule:

Never display vendor names such as:

- Claude
- GPT
- Gemini
- Grok

as visible workflow identities.

Users should experience **roles**, not **vendors**.

---

# 4. Render Layer Visibility Rule (Optional Exposure Layer)

Rendering engines create visibly different outputs.

Because they materially affect results, they may be optionally exposed in controlled ways.

Render layer includes:

- image generation
- video generation
- motion synthesis
- voice generation
- sound generation
- upscaling
- style rendering

Examples of render engines:

- Kling
- Runway
- Ideogram
- Flux
- Segmind engines
- fal.ai engines
- other supported visual/audio engines

Important rule:

Rendering engines should **not dominate the primary UI**, but may be visible inside controlled areas.

---

# 5. UI Presentation Structure

## Default Mode (Beginner)

Keep the interface simple.

Show outcome-based controls instead of vendor names.

Examples:

- Visual Engine: Auto
- Image Engine: Auto
- Audio Engine: Auto
- Quality: Fast / Standard / Premium
- Budget Mode: Cheapest / Balanced / Best Quality
- Style Strength: Basic / Strong
- Character Consistency: On / Off
- Story Intelligence: On

Do NOT force beginners to pick:

- provider brands
- model IDs
- internal routing systems

---

## Advanced Mode (Power Users)

Allow optional advanced access panels.

Examples:

- Engine Details
- Advanced Provider Settings
- Rendering Engine Selection

Advanced users may see:

- Video Engine: Auto / Kling / Runway / others
- Image Engine: Auto / Ideogram / Flux / others
- Audio Engine: Auto / selectable provider
- Budget Mode: Cheapest / Balanced / Best
- Speed Preference: Fast / Balanced / Maximum Quality

Important rule:

Planning providers remain hidden even in advanced mode unless admin-level access is required.

---

# 6. Branding Strategy Rule

## Do Not Lead With Provider Names

Avoid interfaces that begin with:

- choose GPT
- choose Claude
- choose Gemini
- choose Kling

Users do not care about vendor identity.

Users care about:

- output quality
- cost
- speed
- realism
- reliability

---

## Preferred Branding Structure

Show:

Outcome-based naming.

Examples for video:

- Cheapest Preview
- Fast Social Video
- Balanced Story Video
- Premium Performance Video
- Cinematic High Quality

Examples for image:

- Quick Image Draft
- Balanced Ad Image
- Premium Poster Image
- Character Consistency Image

Examples for planning:

- Fast Plan
- Smart Plan
- Producer Plan
- Director Review
- Quality Check

---

# 7. Multi-AI Role Orchestration Rule

GHS may use multiple reasoning systems internally.

However, the UI must present workflow intelligence as coordinated roles.

Example user-facing explanation:

- AI Story Director created the first draft
- AI Technical Director structured the scenes
- AI Quality Reviewer verified continuity

Not:

- Claude wrote this
- GPT reviewed this

---

# 8. Core Caching Rule

Caching improves cost and speed.

Caching must NOT reduce originality.

**Never cache final personalized outputs.**

Only cache reusable intelligence skeletons.

---

# 9. What Should Be Cached

Reusable intelligence components.

Examples:

- genre-based planning structures
- dance movement grammar
- music rhythm pacing templates
- camera grammar rules
- chase scene baseline logic
- dialogue interruption logic
- sound assumption frameworks
- marketplace noise logic
- hallway tension logic
- palace environment patterns
- village ambience templates
- children learning structures
- jingle composition frameworks
- lyric video baseline templates
- visualizer baseline structures

These are reusable planning skeletons.

---

# 10. What Must Never Be Cached

Do NOT cache:

- final storylines
- final scene lists
- user-specific narration
- emotional arcs
- character identity logic
- personalized creative prompts
- final unique pacing

These must always be generated fresh.

---

# 11. Correct Caching Workflow

Correct system flow:

User Input
→ Retrieve Intelligence Skeletons
→ Personalize Using Low-Cost Planning
→ Generate Unique Final Output

Formula:

Cached Blueprint + Live Personalization = Unique Result

---

# 12. Cost Strategy Rule

Planning cost should remain very low.

Rendering cost is the primary expense.

---

## Cheap Models Should Handle

- idea expansion
- technical inference
- first-pass planning
- sound/environment logic
- choreography suggestion
- initial review logic

---

## Premium Models Should Handle

- complex reasoning
- premium-tier workflows
- long-form validation
- difficult continuity cases
- final production-level polish

---

# 13. Credit System Strategy

Users must not see token math.

Users interact through credit units.

Examples:

- Movie Plan Generation → small credit bundle
- Scene Rendering → heavier credit action
- Music Generation → moderate credit action

Credits represent value, not internal computation.

---

# 14. Provider Visibility Principle

Core philosophy:

**Hide the brains. Optionally expose the render machines.**

Meaning:

Hidden layer:

- LLM reasoning
- planning logic
- inference routing

Optional exposure:

- image engines
- video engines
- audio engines

Only expose engines when they materially change output results.

---

# 15. Recommended Default Settings

For most users:

- Story Intelligence: On
- Visual Engine: Auto
- Image Engine: Auto
- Audio Engine: Auto
- Quality Level: Standard
- Budget Mode: Balanced

These settings reduce friction and confusion.

---

# 16. Recommended Advanced Settings

Advanced users may access:

- Video Engine Selection
- Image Engine Selection
- Audio Engine Selection
- Cost Preference
- Speed Preference

Examples:

- Video Engine: Auto / Kling / Runway
- Image Engine: Auto / Ideogram / Flux
- Audio Engine: Auto / Selected Provider
- Cost Preference: Cheapest / Balanced / Highest Quality
- Speed Preference: Fast / Balanced / Best Output

---

# 17. Rendering vs Intelligence Architecture

Separate GHS into two independent layers.

## A. Intelligence Layer (Hidden)

Handles:

- planning
- inference
- interpretation
- technical preparation
- quality scoring
- logic correction

User sees results only.

Never provider identity.

---

## B. Render Layer (Optional Exposure)

Handles:

- image creation
- video generation
- sound generation
- animation logic
- motion synthesis

May be optionally visible in advanced mode.

---

# 18. Product Identity Rule

GHS must never feel like:

A thin wrapper around AI providers.

Instead, it must feel like:

A production system powered by structured intelligence.

Users must trust GHS workflow — not vendor names.

---

# 19. MVP Implementation Policy

For MVP stage:

Do:

- use GHS-branded role names
- hide planning providers
- expose render engines only in advanced mode
- cache reusable skeletons
- personalize final outputs live

Do NOT:

- expose raw model names in default UI
- allow repetitive cached outputs
- confuse users with backend complexity

---

# 20. Final Master Principle

**GHS must present intelligence through branded workflow roles, while exposing rendering engines only when useful. Caching must optimize reusable intelligence — never final personalized output.**

This preserves:

- product clarity
- brand ownership
- flexibility
- cost control
- output originality
