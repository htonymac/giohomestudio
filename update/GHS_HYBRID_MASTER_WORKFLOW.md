## 1. Core Definition of GHS Hybrid Mode

### Product Meaning
GHS Hybrid Mode = Text + Character Intelligence + Scene Planning + Audio + Images + Motion + Video + Controlled Assembly

Hybrid mode is a core GHS production format. It is not a fallback. It is not a side option. It is one of the strongest cost-saving and quality-defining systems inside GHS.

### Hybrid Core Equation
Hybrid Movie = Story Intent + Character Identity Objects + Scene Objects + Shot Objects + Narration Strategy + Dialogue Ownership + Image Scenes + Image-to-Motion Scenes + Video Scenes + Audio Bridge Logic + Timeline Assembly

### Core Hybrid Philosophy
Not every scene should be video. Not every scene should be still. Not every moment should be narrated the same way.

The correct hybrid logic is:
- Images carry setup, emotion, atmosphere, and cheaper storytelling moments
- Motion bursts carry transition, lift, emphasis, and controlled movement
- Video carries action, body movement, reaction, impact, and payoff
- Audio carries continuity, life, immersion, and emotional glue
- Narration changes by scene type and must not be treated the same everywhere

This is the default intelligence rule for Hybrid Mode.

## 2. Global Non-Negotiable System Laws

These rules are mandatory. No hybrid implementation is allowed to bypass them.

### Rule 1 — Characters Are Structured Identity Objects
Characters must never exist only as plain text names inside prompts.
Every character must exist as a structured object with a persistent Character ID.

Character identity must persist across:
- scene planning
- shot planning
- image generation
- video generation
- narration planning
- dialogue assignment
- editing
- regeneration
- timeline assembly
- export

Character names alone are not enough. Character IDs are mandatory.

### Rule 2 — Story Must Always Become Scene Objects
Hybrid stories must never render directly from one long paragraph.

Every story must pass through:
1. story input
2. story expansion
3. scene breakdown
4. scene storage
5. shot planning
6. review state
7. user-controlled assembly

All story output must become structured scene objects before generation.

### Rule 3 — Draft First, Assembly Later
All hybrid scenes must remain in draft state until the user explicitly triggers assembly.
Hybrid Mode must never auto-assemble a full movie by default.

Assembly begins only when the user triggers: **"Assemble My Scenes"**

Before that moment:
- scenes remain separate
- scene types remain editable
- narration remains adjustable
- shot structure remains editable
- image vs video decisions remain editable
- audio plans remain editable
- costs remain inspectable

### Rule 4 — No Local LLM for Core Intelligence
All core hybrid planning must use reliable cloud reasoning models.

Local LLMs must not be used for:
- story planning
- character design
- scene breakdown
- hybrid routing
- dialogue planning
- continuity decisions
- narration logic
- shot planning
- identity consistency

Local models may be used only for low-risk experimental preview helpers, never for core planning authority.

### Rule 5 — Identity Stability Over Prompt Convenience
Prompt shortcuts must never break identity consistency.

Character IDs, voice IDs, scene IDs, and shot IDs must remain stable across:
- retries
- shot regeneration
- scene replacement
- timeline edits
- hybrid upgrades
- collaborative editing

## 6. Step-by-Step Hybrid Pipeline

### Step 1 — User Story Input
The user enters any of: idea, concept, story prompt, script, treatment, narrative summary, commercial idea, children story concept, narrated movie concept.

Optional user controls: target movie length, audience type, language, narration preference, cost preference, quality mode, region/cultural style, preferred hybrid mode.

### Step 2 — Story Expansion
A cloud reasoning model expands the user input into a structured story plan.

Required output: story summary, tone, pacing direction, world logic, emotion logic, narrative arc, character list, location list, sound suggestions, mood suggestions, likely action peaks, likely narration-heavy areas.

### Step 3 — Character Extraction and Character Registry Creation
Every detected character must be converted into a persistent Character Identity Object.

Mandatory Character Identity Fields:
- Character ID
- Display Name
- Role Type
- Gender / Presentation
- Age Range
- Height
- Body Type
- Skin Tone
- Hair Style
- Facial Traits
- Wardrobe Style
- Speech Style
- Accent Type
- Emotion Profile
- Voice ID
- Language
- Character Tags

Example: CH01 = Tunde, CH02 = Zara, CH03 = Narrator

These IDs must be reused everywhere. Never replaced. Never randomized.

### Step 4 — Character Asset Registry
Minimum Required Visual Assets: hero portrait, 3/4 portrait, side profile, full body, expression variant.
Optional: costume variants, lighting variants, mood variants, action pose references.

Reference assets must feed both image generation and video generation.

### Step 5 — Character Voice Registry
Example: CH01 -> Voice_Tunde_Adult_Male_NG, CH02 -> Voice_Zara_Young_Female_NG, CH03 -> Voice_Narrator_Calm_English_African

Dialogue must never receive random voice assignment.

### Step 6 — Scene Breakdown
Mandatory Scene Object Fields: Scene ID, Scene Title, Story Order Index, Location, Time of Day, Weather, Mood, Characters Present (Character IDs), Primary Speaker, Secondary Speakers, Lighting Plan, Camera Style Suggestion, Sound Suggestion, Music Suggestion, Dialogue Map, Scene Duration Estimate, Scene Status, Draft State.

### Step 7 — Hybrid Scene Classification
Mandatory Hybrid Scene Types:
1. Image-Led Scene
2. Video-Led Scene
3. Image-to-Video Scene
4. Audio-Heavy Bridge Scene
5. Hybrid Scene

### Step 8 — Scene Intelligence Scoring
Required Per-Scene Intelligence Fields: Scene Type, Motion Need (low/medium/high), Image Suitability Score, Motion Necessity Score, Recommended Motion Duration, Narration Intensity (low/medium/high), Dialogue Density (low/medium/high), Emotional Weight (low/medium/high), Cost Priority (efficient/balanced/premium), Full Video Justification.

Supporting Tags: narration-heavy, dialogue-heavy, action-heavy, emotion-heavy, transition, montage, reveal, suspense, reflective, payoff.

### Step 9 — Shot Planning Layer
Mandatory Shot Object Fields: Shot ID, Scene ID, Shot Order Index, Characters Visible (IDs), Speaking Character ID, Listening Character IDs, Camera Angle, Camera Movement, Framing Type, Lighting Style, Dialogue Line, Audio Timing, Environment SFX, Planned Media Type, Planned Duration, Regeneration Eligibility Flag.

### Step 10 — Dialogue Ownership Mapping
Every spoken line must map to: Character ID, Voice ID, Scene ID, Shot ID, timing role.

### Step 11 — Narration Strategy Engine
- Image-Led: narration medium to high
- Video-Led: narration low or none
- Image-to-Video: narration tapers as motion takes over
- Audio-Heavy Bridge: narration strongest
- Hybrid: narration adapts per shot

### Step 12 — Audio Planning Layer
Required: dialogue plan, narration plan, ambience plan, weather sound plan, object SFX plan, action SFX plan, music cue plan, silence/intensity control moments, transition audio logic.

Audio Truth Rule: Even image-led scenes need ambience, subtle environment sound, music intention, narration fit.

### Step 13 — Media Routing Engine
Routing decisions: image generation, image with camera movement, parallax/pan/zoom treatment, image-to-video treatment, short motion burst generation, full video generation, audio-only bridge support.

### Step 14 — Visual Generation Workflow
Stage A: Character Reference Confirmation
Stage B: Scene Still Generation (establishing frame, layout preview)
Stage C: Shot-Level Media Generation
