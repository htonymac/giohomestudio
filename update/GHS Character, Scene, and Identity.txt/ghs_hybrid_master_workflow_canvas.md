# 1. Core Definition of GHS Hybrid Mode

## Product Meaning

**GHS Hybrid Mode = Text + Character Intelligence + Scene Planning + Audio + Images + Motion + Video + Controlled Assembly**

Hybrid mode is a core GHS production format.
It is not a fallback.
It is not a side option.
It is one of the strongest cost-saving and quality-defining systems inside GHS.

---

## Hybrid Core Equation

Hybrid Movie =
Story Intent
+ Character Identity Objects
+ Scene Objects
+ Shot Objects
+ Narration Strategy
+ Dialogue Ownership
+ Image Scenes
+ Image-to-Motion Scenes
+ Video Scenes
+ Audio Bridge Logic
+ Timeline Assembly

---

## Core Hybrid Philosophy

Not every scene should be video.
Not every scene should be still.
Not every moment should be narrated the same way.

The correct hybrid logic is:

- **Images** carry setup, emotion, atmosphere, and cheaper storytelling moments
- **Motion bursts** carry transition, lift, emphasis, and controlled movement
- **Video** carries action, body movement, reaction, impact, and payoff
- **Audio** carries continuity, life, immersion, and emotional glue
- **Narration** changes by scene type and must not be treated the same everywhere

This is the default intelligence rule for Hybrid Mode.

# 2. Global Non-Negotiable System Laws

These rules are mandatory.
No hybrid implementation is allowed to bypass them.

## Rule 1 — Characters Are Structured Identity Objects

Characters must never exist only as plain text names inside prompts.

Every character must exist as a structured object with a persistent **Character ID**.

Character identity must persist across:

- scene planning
- shot planning
- image generation
- video generation
- narration planning
- dialogue assignment
- editing
- regeneration
- timeline assembly
- export

Character names alone are not enough.

**Character IDs are mandatory.**

---

## Rule 2 — Story Must Always Become Scene Objects

Hybrid stories must never render directly from one long paragraph.

Every story must pass through:

1. story input
2. story expansion
3. scene breakdown
4. scene storage
5. shot planning
6. review state
7. user-controlled assembly

All story output must become structured scene objects before generation.

---

## Rule 3 — Draft First, Assembly Later

All hybrid scenes must remain in **draft state** until the user explicitly triggers assembly.

Hybrid Mode must never auto-assemble a full movie by default.

Assembly begins only when the user triggers:

**“Assemble My Scenes”**

Before that moment:

- scenes remain separate
- scene types remain editable
- narration remains adjustable
- shot structure remains editable
- image vs video decisions remain editable
- audio plans remain editable
- costs remain inspectable

---

## Rule 4 — No Local LLM for Core Intelligence

All core hybrid planning must use reliable cloud reasoning models.

Local LLMs must not be used for:

- story planning
- character design
- scene breakdown
- hybrid routing
- dialogue planning
- continuity decisions
- narration logic
- shot planning
- identity consistency

Local models may be used only for low-risk experimental preview helpers, never for core planning authority.

---

## Rule 5 — Identity Stability Over Prompt Convenience

Prompt shortcuts must never break identity consistency.

Character IDs, voice IDs, scene IDs, and shot IDs must remain stable across:

- retries
- shot regeneration
- scene replacement
- timeline edits
- hybrid upgrades
- collaborative editing

# 3. Hybrid Mode Purpose and Product Strength

GHS Hybrid Mode exists to make long-form and medium-form storytelling practical, cinematic, and cost-aware.

It allows GHS to:

- reduce full-video generation cost

- keep emotional scenes strong
- use motion only where motion truly matters
- make narration smarter instead of generic
- let audio do real structural work

- support review and controlled assembly like a real production system

Without this system, GHS risks:

- over-spending on video
- random scene treatment
- weak continuity
- narration mismatch
- static-feeling output
- chaotic regeneration

# 6. Step-by-Step Hybrid Pipeline

## Step 1 — User Story Input

The user enters any of the following:

- idea
- concept
- story prompt
- script
- treatment
- narrative summary
- commercial idea
- children story concept
- narrated movie concept

At this stage, GHS captures user intent only.

No final media generation happens here.

The system may also capture optional user controls such as:

- target movie length
- audience type
- language
- narration preference
- cost preference
- quality mode
- region or cultural style
- preferred hybrid mode

---

## Step 2 — Story Expansion

A cloud reasoning model expands the user input into a structured story plan.

Required output should include:

- story summary
- tone
- pacing direction
- world logic
- emotion logic
- narrative arc
- character list
- location list
- sound suggestions
- mood suggestions
- likely action peaks
- likely narration-heavy areas

This stage prepares the story for structured hybrid planning.

---

## Step 3 — Character Extraction and Character Registry Creation

Every detected character must be converted into a persistent **Character Identity Object**.

### Mandatory Character Identity Fields

Each character object must include at minimum:

- Character ID
- Display Name
- Role Type
- Gender / Presentation
- Age Range
- Height
- Body Type
- Skin Tone
- Hair Style
- Facial Traits
- Wardrobe Style
- Speech Style
- Accent Type
- Emotion Profile
- Voice ID
- Language
- Character Tags

Example:

- CH01 = Tunde
- CH02 = Zara
- CH03 = Narrator

These IDs must be reused everywhere.
Never replaced.
Never randomized.

---

## Step 4 — Character Asset Registry

Each character must have approved reference assets stored in a reusable registry.

### Minimum Required Visual Assets

- hero portrait
- 3/4 portrait
- side profile
- full body
- expression variant

Optional but recommended:

- costume variants
- lighting variants
- mood variants
- action pose references
- age or episode variants where valid

Hybrid generation must not keep re-inventing the same character from scratch.

Reference assets must feed both:

- image generation
- video generation

---

## Step 5 — Character Voice Registry

Every speaking character must map to a persistent voice identity.

Example:

- CH01 → Voice_Tunde_Adult_Male_NG
- CH02 → Voice_Zara_Young_Female_NG
- CH03 → Voice_Narrator_Calm_English_African

Dialogue must never receive random voice assignment.

Narrator voice must also be treated as a controlled identity object, not a random afterthought.

---

## Step 6 — Scene Breakdown

The expanded story must be split into structured scene objects.

### Mandatory Scene Object Fields

Each scene must include:

- Scene ID
- Scene Title
- Story Order Index
- Location
- Time of Day
- Weather
- Mood
- Characters Present (Character IDs)
- Primary Speaker
- Secondary Speakers
- Lighting Plan
- Camera Style Suggestion
- Sound Suggestion
- Music Suggestion
- Dialogue Map
- Scene Duration Estimate
- Scene Status
- Draft State

All scenes must be saved immediately after generation.

---

## Step 7 — Hybrid Scene Classification

Every scene must be classified before rendering.

### Mandatory Hybrid Scene Types

1. **Image-Led Scene**
2. **Video-Led Scene**
3. **Image-to-Video Scene**
4. **Audio-Heavy Bridge Scene**
5. **Hybrid Scene**

These are structural categories, not cosmetic tags.

---

## Step 8 — Scene Intelligence Scoring

Each scene must receive hybrid intelligence scores to help routing and user review.

### Required Per-Scene Intelligence Fields

- Scene Type
- Motion Need: low / medium / high
- Image Suitability Score
- Motion Necessity Score
- Recommended Motion Duration
- Narration Intensity: low / medium / high
- Dialogue Density: low / medium / high
- Emotional Weight: low / medium / high
- Cost Priority: efficient / balanced / premium
- Full Video Justification (required when motion need is high)

### Recommended Supporting Tags

- narration-heavy
- dialogue-heavy
- action-heavy
- emotion-heavy
- transition
- montage
- reveal
- suspense
- reflective
- payoff

These supporting tags help hybrid routing, but they do not replace the mandatory scene type.

---

## Step 9 — Shot Planning Layer

No scene should go directly from scene plan into final generation without shot-level planning.

Every hybrid scene must be broken into shots first.

### Mandatory Shot Object Fields

Each shot must include:

- Shot ID
- Scene ID
- Shot Order Index
- Characters Visible (IDs)
- Speaking Character ID
- Listening Character IDs
- Camera Angle
- Camera Movement
- Framing Type
- Lighting Style
- Dialogue Line
- Audio Timing
- Environment SFX
- Planned Media Type
- Planned Duration
- Regeneration Eligibility Flag

Shot planning is critical for:

- multi-character control
- who speaks when
- who is visible when
- which shot is still vs motion vs video
- preserving continuity across mixed formats

---

## Step 10 — Dialogue Ownership Mapping

Dialogue must attach to character IDs, not just text labels.

Example:

- CH01: “You sure say this plan go work?”
- CH02: “If we no move now, we go lose am.”
- CH03: “Night was already pressing in.”

Every spoken line must map to:

- Character ID
- Voice ID
- Scene ID
- Shot ID
- timing role

This prevents:

- wrong speaker
- wrong voice
- mixed dialogue output
- continuity failure during regeneration

---

## Step 11 — Narration Strategy Engine

Narration must change by scene type.

This is one of the most important hybrid rules.

### A. Image-Led Scene Narration

Narration may be stronger here.

Narration can help with:

- describing setting
- explaining emotional state
- implying motion not shown directly
- world-building
- carrying story progression

Narration intensity is usually **medium to high**.

---

### B. Video-Led Scene Narration

Narration should usually be light, emotionally selective, or absent.

Because visible motion already carries meaning, narration must not over-explain obvious action.

Narration intensity is usually **low or none**.

The scene should rely more on:

- action audio
- ambience
- music
- character performance
- visible payoff

---

### C. Image-to-Video Scene Narration

Narration may begin descriptive and then reduce as motion takes over.

This type often benefits from narration tapering.

---

### D. Audio-Heavy Bridge Scene Narration

Narration may be strongest here.

This is where the story may need guidance through:

- scene transition
- unseen events
- time passage
- suspense bridge
- emotional linking

---

### E. Hybrid Scene Narration

Narration must adapt per shot inside the same scene.

For example:

- opening still shot: medium narration
- motion burst: narration drops
- closing reaction: short emotional line

Narration logic must be scene-aware and shot-aware.

---

## Step 12 — Audio Planning Layer

Audio is not decoration.

In Hybrid Mode, audio is structural glue.

Every scene must receive an audio plan before rendering.

### Required Audio Planning Fields

- dialogue plan
- narration plan
- ambience plan
- weather sound plan
- object SFX plan
- action SFX plan
- music cue plan
- silence/intensity control moments
- transition audio logic

### Audio Truth Rule

Even when a scene is image-led, audio must make it feel alive.

That means image scenes still need:

- ambience
- subtle environment sound
- music intention
- narration fit

Audio continuity must flow across scene boundaries.

---

## Step 13 — Media Routing Engine

Once character, scene, shot, narration, and audio plans exist, the system decides what media generation path each shot or scene should take.

### Required Routing Decision Types

- image generation
- image with camera movement
- parallax / pan / zoom treatment
- image-to-video treatment
- short motion burst generation
- full video generation
- audio-only bridge support

### Routing Criteria Must Evaluate

- scene type
- shot type
- motion need
- scene importance
- emotional weight
- dialogue density
- cost mode
- user preference
- model capability

### Example Routing Logic

- image-led setup scene → image model + camera motion + ambience + narration
- emotional close-up with subtle lift → image-to-video
- chase or jump or fight → video model
- dark suspense bridge → audio-heavy bridge with low visual spend
- premium climax → premium video route

Hybrid Mode must not use one video engine blindly for all scenes.

---

## Step 14 — Visual Generation Workflow

### Stage A — Character Reference Confirmation

Before scene generation begins, the system must confirm that all required character references exist.

If missing, they must be generated first and stored.

### Stage B — Scene Still Generation

Every scene should first receive:

- establishing frame
- layout preview
- composition preview

This provides review control before expensive generation.

### Stage C — Shot-Level Media Generation

Only after shots exist should final scene media generation happen.

Supported engines may include:

- Kling
- Wan
- Veo
- Runway
- other future providers

Character references must be attached where identity continuity matters.

# 7. Hybrid Mode Variants Visible to Users

Hybrid Mode may expose guided variants for planning clarity.

## Mode A — Full Video Movie

Most scenes routed to video.
Used when budget is high and constant motion is important.

## Mode B — Hybrid Movie (Core Recommended)

Balanced mix of images, motion, video, and strong audio.
This should be the main recommendation.

## Mode C — Image-Led Narrated Movie

Heavier narration, lighter motion.
Useful for children stories, emotional narration, recap storytelling, and lower-cost output.

## Mode D — Hybrid Cinematic Movie

Selective premium motion bursts, stronger cinematic sound design, stronger payoff routing.

These variants must still obey the same identity and scene workflow rules.

# 9. Assemble My Scenes Rule

Assembly is a separate stage.

It must begin only when the user clicks or triggers:

**Assemble My Scenes**

Before assembly, there is no locked final movie timeline.

### Required Assembly Validation Steps

1. validate scene order
2. validate scene type routing completeness
3. validate character continuity
4. validate voice mapping
5. validate dialogue ownership
6. validate audio mapping
7. validate missing assets
8. validate shot timing
9. validate transition readiness

Only after passing validation should timeline build begin.

---

# 10. Timeline Builder and Finishing Layer

Once assembly begins, GHS creates the hybrid timeline.

### Timeline Builder Responsibilities

- merge approved scenes in order
- place image scenes with motion treatments
- place video scenes accurately
- place audio bridges between scenes where needed
- sync narration to scene timing
- sync dialogue timing to shots
- place ambience and sound layers
- apply music cues
- manage transitions
- lock timing after approval

### Finishing Layer Minimum Controls

Inspired by practical modern editor workflows, the finishing layer must support:

- trim
- cut
- reorder
- subtitle sync
- audio level control
- scene lock
- scene replace
- transition control
- scene mute / narration mute
- shot-level replacement where supported

The finishing layer must not break character IDs, scene IDs, shot IDs, or voice IDs.

---

# 11. Regeneration Rules

Hybrid Mode must support selective regeneration.

## Regeneration Truth Rule

Only the affected scene or shot should regenerate unless the user explicitly requests a wider rebuild.

### Allowed Regeneration Targets

- regenerate shot only
- regenerate image only
- regenerate motion treatment only
- regenerate video shot only
- change narration for one scene
- change voice for one character where permitted
- update one character reference set
- replace music or ambience layer only

### Not Allowed by Default

- random full-timeline rebuilds
- hidden character replacement
- random voice switching
- scene identity reset

---

# 12. Continuity Validation System

Before rendering and before export, the system must automatically check for consistency failures.

## Required Continuity Checks

- wrong speaker
- voice mismatch
- missing character reference
- duplicate character identity
- scene order conflict
- missing shot data
- missing ambience or dialogue mapping
- wardrobe mismatch where continuity matters
- face mismatch across image and video scenes
- character visible in shot but not registered in scene
- dialogue line with no character ownership

Errors must surface clearly before final render.

# 18. Example Hybrid Logic in Practice

A sample hybrid story may behave like this:

### Scene 1 — Mountain Edge Setup

- scene type: image-led
- narration: medium to high
- motion: low
- audio: wind, distant ambience, suspense cue

### Scene 2 — Close-up Before Jump

- scene type: image-to-video
- narration: medium tapering to low
- motion: medium
- audio: breathing, boot movement, wind pressure

### Scene 3 — Jump

- scene type: video-led
- narration: none or very light
- motion: high
- audio: air rush, score lift, action SFX

### Scene 4 — Freefall Bridge

- scene type: hybrid or video-led
- narration: low
- audio: intense wind, emotional score

### Scene 5 — Safe Landing Reflection

- scene type: hybrid or image-led closing beat
- narration: medium
- audio: quieter ambience, emotional release music

This is the kind of scene-aware routing GHS Hybrid Mode must own.
